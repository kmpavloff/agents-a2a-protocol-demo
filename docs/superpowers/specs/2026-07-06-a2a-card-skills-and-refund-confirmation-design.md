# Дизайн: навыки из AgentCard + подтверждение возвратов (HITL)

Дата: 2026-07-06

## Контекст и проблема

Демо показывает протокол A2A: пользователь общается с **оркестратором**, который делегирует
работу с заказами удалённому **воркеру** (`orders-agent`) через A2A. Сейчас есть два разрыва
между тем, «как есть», и тем, «как правильно» по духу A2A/adk:

1. **Оркестратор хардкодит знание о воркере.** `orchestratorInstruction`
   (`internal/agent/orchestrator.go`) вручную описывает домен воркера («заказы, статусы,
   статистика, возвраты», примеры), а `OrdersClient.Tool()` (`internal/a2abridge/client.go`)
   хардкодит имя `ask_orders_agent` и его описание. При этом `NewOrdersClient` **уже резолвит**
   `AgentCard` воркера, но использует из неё только `card.Name` для трейса — `card.Skills`
   игнорируется. Смысл A2A ровно в том, чтобы клиент **узнавал** возможности агента из карточки.

2. **Возврат средств выполняется без подтверждения человека.** Воркер может вызвать
   `initiate_refund` и оформить возврат молча. Нужен барьер Human-in-the-Loop.

Обе задачи — про правильное использование A2A/adk вместо хардкода, поэтому решаются одной спекой.

## Цели

- Оркестратор строит имя инструмента, его описание и доменный блок своего промпта **из AgentCard**
  воркера, полученной по A2A. Домен «заказы» исчезает из кода оркестратора.
- Воркер обогащает свой единственный навык `manage_orders` полями `Tags` и `Examples`.
- Возврат (`initiate_refund`) требует подтверждения человека через нативный HITL adk,
  переданный человеку по существующему каналу A2A `input-required`.

## Не-цели (YAGNI)

- Несколько навыков у воркера (оставляем один `manage_orders`, обогащённый).
- Конфигурируемый список инструментов, требующих подтверждения (подтверждаем только возврат).
- Подтверждение read-only операций.
- Изменения в логике `ask`/pending и трейсинге, кроме необходимых для подтверждения.
- Изменения кода оркестратора/клиента ради подтверждения (переиспользуем `input-required`).

## Компонент 1. Навыки из AgentCard

### 1.1 Обогащение карточки воркера

`a2abridge.AgentCard` (`internal/a2abridge/server.go`). Навык `manage_orders` дополняется:

- `Tags`: `["заказы", "статусы", "статистика продаж", "возвраты", "поиск"]`
- `Examples`: `["верни деньги за заказ 1041", "статус заказа 1041",
  "последние заказы alice", "статистика продаж за 2026-06"]`
- `Description` — чуть детальнее, чем сейчас (сохранить смысл: поиск, статусы, статистика, возвраты).

### 1.2 Деривация профиля (новый `internal/a2abridge/profile.go`)

Чистые функции (легко тестируются изолированно), без побочных эффектов:

```go
type WorkerProfile struct {
    ToolName string // "ask_" + sanitize(card.Name)         → "ask_orders_agent"
    ToolDesc string // из card.Description + навыки + примеры + хвост про NEEDS_USER_INPUT
    Summary  string // человекочитаемый блок «Агент умеет: …» для промпта оркестратора
}

func ProfileFromCard(card *a2a.AgentCard) WorkerProfile
```

- `sanitizeToolName("orders-agent") → "ask_orders_agent"`: не-буквенно-цифровые символы → `_`,
  схлопнуть повторы `_`, префикс `ask_`. Результат должен матчить `^[a-zA-Z_][a-zA-Z0-9_]*$`
  (требование к именам функций-инструментов). Если `card.Name` пуст — запасное имя `ask_agent`.
- `ToolDesc`: `card.Description`, затем по каждому навыку — `Name` + `Description` + список
  `Examples`, затем неизменяемый хвост: «Если он вернёт NEEDS_USER_INPUT, задайте пользователю
  этот вопрос, затем снова вызовите инструмент с его ответом.»
- `Summary`: блок для системного промпта — «Агент по имени <Name> умеет: <навык> — <описание>.
  Примеры запросов: …». Пустые `Examples`/`Skills` не должны ломать сборку (просто опускаются).

### 1.3 Проводка

- `NewOrdersClient` (`client.go`): после резолва карточки строит и **хранит**
  `profile WorkerProfile`; добавить геттер `func (c *OrdersClient) Profile() WorkerProfile`.
- `OrdersClient.Tool()`: имя и описание берутся из `c.profile.ToolName` / `c.profile.ToolDesc`
  вместо литералов (заменяет `client.go:148-151`).
- `NewOrchestrator(model, ordersTool, workerSummary string)`
  (`internal/agent/orchestrator.go`): инструкция становится доменно-нейтральным шаблоном,
  собираемым через `fmt.Sprintf` из:
  - имени инструмента — `ordersTool.Name()`;
  - `workerSummary` — что умеет воркер.
  Поведенческие правила протокола (копировать детали дословно; никогда не слать пустой `message`;
  обработка `NEEDS_USER_INPUT`; сообщать результат сразу, не говорить «сейчас сделаю»; отвечать
  кратко по-русски) **остаются** в шаблоне — они про поведение оркестратора, а не про домен.
- `cmd/orchestrator/main.go`: передаёт `oc.Profile().Summary` в `NewOrchestrator`.

## Компонент 2. Подтверждение возвратов (HITL)

### 2.1 Ключевая идея

Подтверждение — это частный случай уже существующего `input-required`. Канал «воркер спрашивает
человека через оркестратора» уже построен для `NEED_INPUT`. Нативный HITL adk
(`functiontool.Config.RequireConfirmation*` + пакет `tool/toolconfirmation`) подключается к этому
каналу. Барьер детерминированный: без ответа человека возврат физически не выполняется —
в отличие от «попросить модель самой спросить», что маленькая модель может проигнорировать.

Механизм adk:
1. У инструмента задан провайдер подтверждения → adk не выполняет инструмент сразу.
2. При вызове инструмента adk эмитит спец-`FunctionCall` с именем
   `toolconfirmation.FunctionCallName` (`"adk_request_confirmation"`). В его `Args` лежит
   `originalFunctionCall` (имя + аргументы исходного вызова).
3. Приложение отвечает `FunctionResponse{Name: "adk_request_confirmation", ID: <тот же>,
   Response: {"confirmed": bool}}` → adk до-выполняет или отклоняет исходный инструмент.

Шаги 2–3 происходят внутри цикла воркера (`executor.Execute`), а человек — на стороне
оркестратора; мост между ними — A2A `input-required`.

### 2.2 Инструмент (`internal/orders/tools.go`)

`initiate_refund` получает:

```go
RequireConfirmationProvider: func(a idArgs) bool { return a.orderID() != "" }
```

Подтверждаем только когда указан реальный номер заказа — пустой «проверочный» вызов модели
не порождает подтверждения (просто вернётся подсказка «Не указан номер заказа»). Остальные
инструменты read-only — без изменений.

### 2.3 Воркер-executor (`internal/a2abridge/server.go`) — вся новая логика здесь

Executor получает поле состояния (по аналогии с `pending` у клиента, под мьютексом):

```go
pendingConfirm map[string]string // contextID → callID вызова adk_request_confirmation
```

**Обнаружение (в цикле `runner.Run`):** ловим `part.FunctionCall` с
`Name == toolconfirmation.FunctionCallName`. Через `toolconfirmation.OriginalCallFrom(fc)`
достаём исходный вызов (`initiate_refund`, `order_id=1041`) и строим человекочитаемый вопрос,
напр.: «Подтвердите оформление возврата по заказу 1041? (да/нет)». Сохраняем `fc.ID`
в `pendingConfirm[contextID]`, эмитим `input-required` с вопросом и `return` — **артефакт не
эмитим**. (Формируем вопрос сами из `originalFunctionCall`, не полагаясь на англоязычный
`Hint` adk по умолчанию.)

**Возобновление (`ec.StoredTask != nil`):** в начале `Execute` проверяем
`pendingConfirm[contextID]`.
- Если запись есть → это ответ на подтверждение. Детерминированно парсим текст пользователя
  (`да/yes/подтверждаю/ок/да, оформляй` → `confirmed:true`; иначе `false`), удаляем запись,
  строим `genai.Content{Role: user, Parts: [{FunctionResponse: {Name:
  "adk_request_confirmation", ID: callID, Response: {"confirmed": …}}}]}` и подаём его
  в `runner.Run` **вместо** `genai.NewContentFromText(userText, …)`. adk до-выполняет (или
  отклоняет) возврат → далее обычный путь: финальный текст → artifact + completed.
- Если записи нет → существующий путь (обычный текст, в т.ч. возобновление после `NEED_INPUT`).

Трейс дополняется строками про эмит подтверждения и про решение пользователя.

### 2.4 Оркестратор / клиент — без изменений

`input-required` уже превращается клиентом в `NEEDS_USER_INPUT: <вопрос>`
(`client.go:103-110`), оркестратор по существующему правилу промпта задаёт вопрос пользователю
и пересылает ответ. Подтверждение течёт по тому же каналу, что и уточнения.

## Поток данных (сценарий возврата)

1. Пользователь → оркестратор: «верни деньги за заказ 1041».
2. Оркестратор → `ask_orders_agent(message="Оформить возврат по заказу 1041")` → A2A SendMessage.
3. Воркер: LLM вызывает `initiate_refund(order_id=1041)`; adk перехватывает (провайдер вернул
   true) и эмитит `adk_request_confirmation`.
4. Executor сохраняет `callID`, эмитит `input-required` «Подтвердите … 1041? (да/нет)».
5. Клиент → оркестратору `NEEDS_USER_INPUT: Подтвердите … 1041? (да/нет)`; оркестратор
   спрашивает пользователя.
6. Пользователь → оркестратор: «да» → `ask_orders_agent(message="да")` → A2A (тот же task).
7. Executor видит `pendingConfirm[contextID]`, парсит «да» → `confirmed:true`, подаёт
   `FunctionResponse` в runner; adk выполняет возврат.
8. Финальный артефакт «Возврат по заказу 1041 оформлен (…)» → оркестратор → пользователь.

При ответе «нет» (шаг 6): `confirmed:false`, adk блокирует инструмент, воркер сообщает, что
возврат не оформлен; стор не меняется.

## Обработка ошибок и краевые случаи

- **Пустой/проверочный вызов возврата** (нет `order_id`): провайдер возвращает false,
  подтверждение не запрашивается, инструмент возвращает подсказку — как сейчас.
- **`OriginalCallFrom` вернул ошибку**: логируем в трейс, продолжаем как обычный ход
  (не подвешиваем задачу).
- **Неоднозначный ответ пользователя**: любой не-утвердительный ответ трактуется как отказ
  (безопасно по умолчанию — деньги не возвращаются без явного «да»).
- **Пустой `card.Name` / отсутствие навыков**: деривация даёт запасное имя `ask_agent`
  и не падает; `Summary`/`ToolDesc` собираются из доступных полей.
