# A2A Orders Assistant — обучающий проект (дизайн)

**Дата:** 2026-06-22
**Статус:** утверждён к реализации

## Цель

Обучающий проект, демонстрирующий протокол **A2A (Agent2Agent)** на примере двух
агентов на Go:

- **orchestrator** — агент, общающийся с пользователем через минимальный TUI/REPL;
- **worker (orders-agent)** — ведомый агент, выполняющий работу через инструменты
  и доступный по A2A.

Проект должен наглядно показывать три вещи:

1. **Межагентное взаимодействие по A2A** (orchestrator как A2A-клиент → worker как
   A2A-сервер).
2. **Вызовы инструментов** (worker вызывает мок-инструменты «системы заказов»).
3. **Уточнения у пользователя** через каноничное A2A-состояние задачи
   `input-required` с последующим возобновлением той же задачи.

LLM — локальная, запускается пользователем в **LM Studio** (OpenAI-совместимый API).
Проект запускается через **docker compose**.

## Стек

- Язык: **Go**.
- Агенты: **`github.com/google/adk-go`** (LlmAgent, инструменты, цикл LLM).
- Транспорт A2A: **`github.com/a2aproject/a2a-go`** (v2) — A2A-сервер
  (`a2asrv.AgentExecutor`) и A2A-клиент (`a2aclient.Client`).
- Модель: OpenAI-совместимый клиент, указывающий на LM Studio
  (`base_url`, `model`, `api_key`).

> Примечание: точные имена пакетов/конструкторов adk-go (модель под
> OpenAI-совместимый endpoint, обёртка remote-агента как инструмента) уточняются на
> этапе плана реализации через актуальную документацию.

## Архитектура

```
┌─────────────────────────┐   A2A (JSON-RPC + SSE)   ┌──────────────────┐
│      orchestrator        │ ───────────────────────► │     worker       │
│  TUI (REPL) + adk-go LLM │ ◄─────────────────────── │  orders-agent    │
│      A2A-клиент          │   input-required / done  │  adk-go LLM      │
└────────────┬─────────────┘                          │  A2A-сервер      │
             │                                         └────────┬─────────┘
             └──────────► LM Studio (OpenAI-compat) ◄───────────┘
                          http://host.docker.internal:1234/v1
                          мок-инструменты ↑ (data/orders.json)
```

### orchestrator

- Единственная точка общения с пользователем — минимальный **TUI/REPL**: строчный
  цикл «ввод пользователя → ответ агента», с цветными подсказками для статусов
  (работает / ждёт уточнения / ошибка).
- Внутри — **LlmAgent (adk-go)**, ходящий в LM Studio.
- Удалённый worker подключён к оркестратору **как инструмент** `ask_orders_agent`,
  реализация которого под капотом делает A2A-вызов к worker'у через
  `a2aclient.Client`.
- Хранит состояние «ожидающей» A2A-задачи (`taskId`/`contextId`), чтобы возобновить
  ту же задачу после получения уточнения от пользователя.

### worker (orders-agent)

- **A2A-сервер** на `a2aproject/a2a-go` с агентской карточкой (Agent Card),
  описывающей навыки.
- Внутри — **LlmAgent (adk-go)** с набором мок-инструментов работы с заказами.
- При нехватке данных для выполнения запроса возвращает A2A-задачу в состоянии
  **`input-required`** с текстом вопроса; после получения ответа (новое сообщение в
  той же задаче) продолжает работу.

## Сквозной поток (пример «возврат денег»)

1. Пользователь в TUI: «хочу вернуть деньги за последний заказ».
2. orchestrator-LLM вызывает инструмент `ask_orders_agent(...)` → A2A `SendMessage`
   к worker.
3. worker-LLM вызывает `list_recent_orders`, видит неоднозначность → возвращает A2A
   **`input-required`**: «Нашёл #1023 и #1041, какой вернуть?».
4. orchestrator получает `input-required`, **сохраняет `taskId`/`contextId`**
   worker'а и задаёт вопрос пользователю в TUI.
5. Пользователь: «#1041» → orchestrator возобновляет **ту же** A2A-задачу,
   отправляя ответ сообщением с теми же `taskId`/`contextId`.
6. worker-LLM вызывает `initiate_refund(#1041)` → A2A-задача переходит в
   `completed` с артефактом-результатом.
7. orchestrator-LLM формирует финальный ответ → печать в TUI.

Уточнение у пользователя живёт целиком внутри оркестратора (TUI). Демонстрируется
именно состояние A2A `input-required` и возобновление задачи.

## Мок-инструменты worker'а (`internal/orders`)

In-memory store, засеянный из `data/orders.json` при старте. Инструменты:

| Инструмент | Назначение |
|---|---|
| `find_order(query)` | Поиск заказа по тексту/идентификатору |
| `get_order_status(id)` | Статус конкретного заказа |
| `list_recent_orders(customer)` | Последние заказы клиента |
| `get_sales_stats(period)` | Сводная статистика продаж за период |
| `initiate_refund(id)` | Оформление возврата по заказу |

Ошибки предметной области (заказ не найден, возврат недоступен) возвращаются как
обычный результат инструмента — их интерпретирует LLM, а не падение программы.

## Конфигурация (файлы)

YAML на каждый сервис; значения переопределяются переменными окружения (для docker).

- `configs/orchestrator.yaml`: адрес worker'а (A2A URL), `llm.{base_url, model, api_key}`.
- `configs/worker.yaml`: адрес прослушивания A2A, `llm.{base_url, model, api_key}`,
  путь к `data/orders.json`.

Валидация обязательных полей на старте каждого сервиса.

## Обработка ошибок

- **LM Studio недоступен** → понятное сообщение пользователю в TUI (для оркестратора)
  и в лог (для worker'а), без паники.
- **Worker недоступен из оркестратора** → инструмент `ask_orders_agent` возвращает
  ошибку, оркестратор отвечает пользователю мягко.
- **Ошибки инструментов** (заказ не найден и т.п.) → обычный результат инструмента,
  обрабатывает LLM.
- **Протокольные ошибки A2A** → корректные ошибки A2A.
- **Невалидный конфиг** → завершение с понятным сообщением на старте.

## Тестирование (TDD)

- **Unit:**
  - инструменты `internal/orders` — table-driven тесты на каждый инструмент,
    включая ветки ошибок;
  - загрузка/валидация конфига.
- **Integration:**
  - worker A2A-сервер поднимается in-process, A2A-клиент шлёт сообщение, проверяется
    полный раунд-трип, включая переход в `input-required` и возобновление задачи;
  - используется **stub-модель** (детерминированная заглушка LLM), чтобы тесты не
    зависели от LM Studio и проходили в CI.

## Docker Compose

- Сервисы: `worker` и `orchestrator` (`depends_on: worker`).
- `orchestrator` запускается интерактивно для TUI: `docker compose run --rm orchestrator`.
- LM Studio работает на хосте; контейнеры обращаются к нему через
  `host.docker.internal` (с `extra_hosts` для Linux).
